# MCP Task Flow

本文档定义每个任务从生成到发布的标准流程。

## 标准任务流

```text
女娲 / CAIO
  -> 诸葛亮 / PM-001
    -> 任务生成
      -> 墨子 / ARC-001 确认架构边界
        -> 专业 Codex 执行
          -> 赵云 / AUD-001 审计
            -> PASS 发布
            -> FAIL 二次复审
              -> PASS 发布
              -> FAIL 退回重新执行
```

## 任务生成规则

任务必须来自以下来源之一：

1. 已确认 PRD
2. 已确认架构文档
3. 已确认数据治理计划
4. 已确认接口协议
5. 已确认审计整改项
6. King Xu 或女娲直接指令
7. 女娲发起的 OMCF 规则维护任务
8. 赵云发起的审计整改任务

## 任务拆分规则

任务拆分时必须满足：

1. 单任务有明确责任 Codex。
2. 单任务输出可审计。
3. 单任务依赖关系清楚。
4. 单任务验收标准可验证。
5. 单任务失败不会模糊影响范围。
6. 命中专家条件时必须添加专家协作角色。

## 专家角色触发规则

任务拆分时如命中以下条件，诸葛亮必须把对应专家加入协作 Codex：

| 专家 | 触发条件 | 必须验证 |
|---|---|---|
| 庞统 / AOEM-001 | AOEM 内核、GPU 调度、隐私计算、代数执行、运行时优化 | AOEM 核心逻辑审计、GPU_dispatch_test、技术复核 |
| 华佗 / MATH-001 | 代数结构、数学证明、索尔算法、量子计算、模型数理验证 | math-model-audit、误差边界、二次确认 |
| 庄周 / LANG-001 | 新语言语义、DSL、指令集、编译优化、自动演化 | lang-design-audit、compiler-sanity-test、墨子审批 |

## 任务依赖图规则

任务依赖图应按以下顺序生成：

1. 文档任务
2. 架构任务
3. 数据任务
4. 接口任务
5. UI 任务
6. 后端任务
7. 前端任务
8. 移动端任务
9. AI 任务
10. AOEM 专项任务
11. 数学模型任务
12. 新语言任务
13. 测试任务
14. 审计任务
15. 发布任务

## FAIL 处理规则

审计失败必须给出：

1. 失败原因
2. 风险级别
3. 影响范围
4. 整改建议
5. 二次复审标准
6. 是否阻断发布

## 发布前确认

发布前必须确认：

1. 任务卡状态为 `AUDIT_PASS`
2. 相关文档已经更新
3. 测试结果可追溯
4. 审计报告已归档
5. 发布与回滚负责人明确
6. 命中人工确认红线的任务已经得到 King Xu 确认
